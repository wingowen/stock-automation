## 4. 书中可提取的策略清单

### 4.1 吸筹阶段

| 事件 | 可量化检测要点 | 难度 |
| --- | --- | --- |
| **PS** (Preliminary Support) | 区间内：放量 + 长下影 + 收高 | 中 |
| **SC** (Selling Climax) | 巨量 + 宽幅阴 + RSI<20 + OBV 跳升 | 易 |
| **AR** (Automatic Rally) | SC 后 1-3 日反弹 ≥50% SC 跌幅 | 易 |
| **ST** (Secondary Test) | AR 高点附近：低量 + 窄区间 + 收高 | 中 |
| **Spring**（弹簧效应） | 跌破下沿 → 1-3 K 收回 + 爆量 | 中 |
| **Test** | 二次测 Spring 底：量更小、收更高 | 易 |
| **SOS** (Sign of Strength) | 放量 + 大阳 + 突破阻力 | 易 |
| **LPS** (Last Point of Support) | 回调：低量 + 小阴 + 接近 SOS 起涨点 | 中 |
| **BU** (Back-Up) | 突破后回踩 + 不破阻力 | 中 |
| **JAC** (Jump Across the Creek) | 放量跳空或连续大阳穿越阻力 | 易 |

### 4.2 派发阶段

| 事件 | 可量化检测要点 | 难度 |
| --- | --- | --- |
| **BC** (Buying Climax) | 巨量 + 长上影 + 高位宽幅 | 易 |
| **UTAD** (Upthrust After Distribution) | 突破新高后快速回落 + 巨量 | 中 |
| **SOW** (Sign of Weakness) | 放量 + 大阴 + 跌破支撑 | 易 |
| **LPSY** (Last Point of Supply) | 反弹：低量 + 小阳 + 不破前高 | 中 |
| **Test** | 高位回测：量减、价高 | 易 |

### 4.3 复合事件

| 事件 | 检测要点 |
| --- | --- |
| **Shakeout** | Spring 后二次跌破 + 量比 Spring 大 |
| **No SC variant** | 直接进入震荡吸筹，无 SC |
| **Cause & Effect** | 区间宽 → 突破后目标位 = 区间高 × 系数 |
| **Effort vs Result** | 高量窄幅 / 低量长 K → 反向信号 |

### 4.4 辅助工具

看盘顺序、9 大买入测试（需读英文原著补全）、9 大卖出测试。

## 5. 推荐实施路线（技能驱动版，Phase A-H）

> **总周期**：约 6-8 周进入模拟盘，3-6 个月才考虑极小仓位实盘。
>
> **每阶段三个视角**：
> - 🎯 **目标**（业务产出）
> - 🛠️ **调用技能**（用哪个 skill 怎么用）
> - ✅ **退出标准**（什么算"做完了"）

### Phase A · 写规格（1-2 天）— **当前阶段**

- 🎯 **目标**：写 1-2 页 `威科夫交易系统/SPEC.md`，让团队对"做什么"达成共识
- 🛠️ **技能调用**：
  1. `interview-me` — 一次一个问题，澄清 4-6 个关键模糊点（成功标准、目标用户、数据频率、止损规则、最大回撤容忍度等）
  2. `spec-driven-development` — 用该 skill 的输出模板写 SPEC.md
  3. `source-driven-development` — 验证 Baostock/Backtrader 选型，引用官方文档
- ✅ **退出标准**：
  - SPEC.md 已合并到 `威科夫交易系统/SPEC.md`
  - SPEC.md 中所有技术选型都有官方文档链接
  - 至少有 1 个 PR review 通过

**反模式**：直接开始写代码；先写 spec 才写代码。

### Phase B · 第一个检测器 TDD 闭环（1-2 天）

- 🎯 **目标**：Spring 检测器 + 测试 + 在 600519 真实数据上跑通
- 🛠️ **技能调用**：
  1. `test-driven-development` — 先写 3 个失败测试（人造数据 + 真实数据 + 边界）
  2. `incremental-implementation` — 一次只让一个测试通过
  3. `verification-before-completion` — 提交前必须确认 `pytest -v tests/test_spring.py` 全部绿
  4. `git-workflow-and-versioning` — 单独 feature 分支
- ✅ **退出标准**：
  - `wyckoff/detectors/spring.py` + `tests/test_spring.py` 提交
  - 真实 600519 数据上 Spring 检测 ≥ 3 个
  - PR 包含 1 张真实数据可视化（matplotlib）

**反模式**：跳过测试写实现；一次写完多个检测器；不验证就合并。

### Phase C · 完整检测器集（1-2 周）

- 🎯 **目标**：完成 §4 的 16+ 形态检测器
- 🛠️ **技能调用**：
  1. `incremental-implementation` — 每天 1 个检测器（重复 Phase B 流程）
  2. `doubt-driven-development` — 每个检测器完成后做 fresh-context 审查
  3. `source-driven-development` — 引用 R. Wyckoff / WyckoffTradingAgent / smartchart 的形态定义
- ✅ **退出标准**：
  - 16+ detector 全部有测试
  - 每个 detector 单独 1 个 PR + 1 张图
  - `doubt-driven-development` 审查无 critical

**反模式**：批量写完所有 detector 再统一测试；不画图验证。

### Phase D · 阶段机与冲突仲裁（3-5 天）

- 🎯 **目标**：A→E 阶段机（FSM）+ 多检测器信号合并 + 冲突解决
- 🛠️ **技能调用**：
  1. `spec-driven-development` — 先写 FSM 规格（状态、转移、守卫）
  2. `test-driven-development` — 状态转移测试 + 冲突仲裁测试
  3. `code-simplification` — 完成后做"减法重构"
  4. `code-review-and-quality` — 5 维审查（正确性/可读性/可测试性/性能/安全）
- ✅ **退出标准**：
  - `wyckoff/phase.py` + 测试
  - 状态机覆盖率 ≥ 95%
  - 5 维审查报告无 critical 问题

**反模式**：直接写 FSM 不写规格；不测冲突场景。

### Phase E · 回测与统计（1 周）

- 🎯 **目标**：沪深 300 成分股 5 年回测，输出夏普/胜率/最大回撤
- 🛠️ **技能调用**：
  1. `test-driven-development` — 策略组合测试（在写策略前）
  2. `performance-optimization` — 检测器用向量化（pandas/numpy）避免逐 K 循环
  3. `verification-before-completion` — 跑回测前必须 `pytest -m strategy` 通过
  4. `documentation-and-adrs` — 回测结果用 ADR 记录（关键决策：为什么选这个组合）
- ✅ **退出标准**：
  - 胜率 ≥ 50%，盈亏比 ≥ 1.5，最大回撤 ≤ 15%，年化夏普 ≥ 1.0
  - **回测样本包含至少 1 轮牛熊**（如 2018-2024）
  - ADR 记录写入 `docs/adr/0001-backtest-methodology.md`

**反模式**：用 1 年数据回测（过拟合风险）；不区分样本内/外。

### Phase F · 看板集成（1 周）

- 🎯 **目标**：把 wyckoff_events 接入 stockexpert-daily-brief 看板
- 🛠️ **技能调用**：
  1. `api-and-interface-design` — 设计 wyckoff_events JSON schema
  2. `frontend-ui-engineering` — 看板新增"威科夫信号池"卡片（注意 WCAG）
  3. `ci-cd-and-automation` — GitHub Actions 加 `wyckoff_scan.py` 任务
  4. `verification-before-completion` — CI 绿 + 本地预览确认
- ✅ **退出标准**：
  - 看板每天自动更新威科夫信号
  - JSON schema 文档化
  - UI 评审通过

**反模式**：直接改现有看板不写 schema；CI 不跑测试就合并。

### Phase G · 模拟盘（4-8 周）

- 🎯 **目标**：跟踪 30+ 模拟交易，验证"信号→决策→结果"链路
- 🛠️ **技能调用**：
  1. `observability-and-instrumentation` — 每日"应买/应卖"日志 + 模拟成交
  2. `verification-before-completion` — 每周自检"信号解释 vs 实际表现"
  3. `documentation-and-adrs` — 每月一次 ADR 记录"为什么调整参数"
  4. `debugging-and-error-recovery` — 信号失败时系统化定位根因
- ✅ **退出标准**：
  - 连续 3 个月模拟盘不亏损
  - 月度 ADR 完整
  - 信号延迟/滑点记录

**反模式**：只跑不记；记了不复盘。

### Phase H · 极小仓位实盘（3-6 月，**最危险**）

- 🎯 **目标**：0.5% 仓位起步，分阶段放量
- 🛠️ **技能调用**：
  1. `security-and-hardening` — API 密钥、交易凭证、日志
  2. `shipping-and-launch` — 上线 checklist + 回滚方案 + 监控告警
  3. `doubt-driven-development` — **每周**对抗式审查"这笔单该不该开"
  4. `verification-before-completion` — 任何改参数必须先回测 + 模拟盘验证
- ✅ **退出标准**：
  - 前 3 个月：单笔风险 ≤ 0.5%，每月 ≤ 3 笔
  - 6 个月：年化夏普 ≥ 1.0、最大回撤 ≤ 10%
  - **模拟盘 vs 实盘差距 ≤ 30%**

**反模式**：跳过模拟盘直接实盘；改参数不重测；不止损。

| 候选 | 推荐 |
| --- | --- |
| **Backtrader** | ✅ **首选**（事件驱动，可对接券商） |
| **VectorBT** | ✅ 辅助（参数扫描） |
| NautilusTrader | ❌ 学习曲线陡 |
| Zipline-reloaded | ❌ 社区冷清 |

### 7.3 检测架构

> **必须用 smartchart 的"Python 检测 + LLM 解释"分层**：
>
> ```
> OHLCV → 指标 (ATR/rvol/摆动) → 规则检测器 → 阶段机 → 冲突仲裁
>       → LLM (只解释，不检测) → 看板 JSON
> ```
>
> **绝对不要让 LLM 直接做检测**（已被验证为 "garbage"）。

### 7.4 风控铁律

1. 单笔最大风险 ≤ 总资金 1%（Phase H 起步时 ≤ 0.5%）
2. 同时持仓 ≤ 5 只
3. 止损硬性：跌破入场 K 低点即出
4. 不抄底、不追涨
5. 月最大回撤 10% 强制停手 1 周
6. 回测样本必须含至少 1 轮牛熊

## 8. 风险与边界声明

> ⚠️ **本报告不是投资建议。**

1. 书中形态是定性描述，无数值化阈值，需自己定
2. LLM 直接做形态检测**会幻觉**，必须"Python 先、LLM 后"
3. A 股 T+1/涨跌停/退市/停牌/ST 必须显式建模
4. 策略未来会失效——聪明钱行为模式变了，策略也会失效
5. 孟洪涛版是入门简化版，**真正 VSA / 9 大买卖测试**需读英文原著
6. 实盘最危险的是自己——务必预设规则、机械执行

### 11.1 强制调用规则

| 场景 | 必须调用的技能 |
| --- | --- |
| 任何新功能/项目开始前 | `using-agent-skills` → 通常落到 `interview-me` 或 `spec-driven-development` |
| 写代码前 | `spec-driven-development`（写规格） |

### 11.2 反模式（禁止）

- ❌ **跳过 SPEC 直接写代码**（导致 Phase 1+ 才发现方向错）
- ❌ **跳过测试写实现**（导致 Phase E 回测基于未验证的检测器）
- ❌ **不画图就合并检测器**（检测器对不对，90% 靠眼看）
- ❌ **批量写完所有 detector**（应一次一个）
- ❌ **直接问 LLM "这是不是 Spring"**（必幻觉）
- ❌ **跳过模拟盘直接实盘**（书 + 现有评估报告都明确警告）
- ❌ **改参数不重新回测**（任何参数调整都要走完整 E 阶段）
