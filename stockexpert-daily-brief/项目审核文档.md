# StockExpert 每日看板自动化（stockexpert-daily-brief）— 项目审核文档

> 文档目的：供审核方全面了解项目的技术架构、GitHub Actions 运行流程、选股/选板块/判断策略，以及数据质量与风险点。
> 审核基准日：2026-07-17（项目实际已生成 2026-07-16、2026-07-17 两份看板数据）。
> 项目定位：把 StockExpert 的「轻量看板模式」搬到 GitHub Actions，每日收盘后自动抓取公开 Web 财经信息，调用 Agnes AI 做综合研判，生成 `看板.json` 并部署到 GitHub Pages，零成本、无本地运行依赖。

---

## 一、总体架构

### 1.1 系统拓扑

```
┌─────────────────────────────────────────────────────────────┐
│  GitHub Actions (ubuntu-latest)                              │
│  触发器: 每个交易日 北京时间 16:30 (UTC 08:35) / 手动触发     │
│                                                             │
│  ┌─────────────── web_brief.py (Python 3.11, 仅标准库) ───┐ │
│  │  1. 交易日历过滤 (周末 + 内置节假日集合)               │ │
│  │  2. build_web_context() 抓取公开 Web 财经信息          │ │
│  │       ├─ 腾讯行情接口 qt.gtimg.cn (指数, 优先)         │ │
│  │       ├─ 东财 push2 接口 (指数, 备用)                 │ │
│  │       ├─ 新浪行业板块名录                              │ │
│  │       ├─ 东财盘中快讯接口                              │ │
│  │       └─ 博查 Bocha Web 搜索 (收评/题材/涨跌家数)     │ │
│  │  3. call_agnes() 调用 Agnes AI (OpenAI 兼容) 做研判   │ │
│  │  4. 用模型给出的个股 code 反查腾讯实时行情(回填硬数字) │ │
│  │  5. assemble() 组装符合 dashboard-contract 的看板.json │ │
│  │  6. write_manifest() 生成 manifest.json               │ │
│  └─────────────────────────────────────────────────────────┘ │
│       │ 落盘: dashboard/data/YYYY-MM/YYYY-MM-DD/看板.json      │
│       │ 落盘: dashboard/manifest.json                          │
│       ▼                                                      │
│  git commit + push (github-actions[bot])                     │
│       ▼                                                      │
│  GitHub Pages 部署 (dashboard/ 整目录)                       │
└─────────────────────────────────────────────────────────────┘
        ▼
┌─────────────────────────────────────────────────────────────┐
│  用户浏览器  →  dashboard/index.html (纯静态, 无后端)        │
│       fetch manifest.json → 左侧日期列表                    │
│       fetch 看板.json    → 右侧大盘/题材/中军/数据质量      │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 分层职责

| 层 | 组件 | 职责 | 技术 |
|---|---|---|---|
| 调度层 | GitHub Actions `daily-brief.yml` | 定时/手动触发，注入 Secrets，跑脚本，提交数据，部署 Pages | YAML + Actions |
| 采集层 | `build_web_context()` | 抓取多源公开 Web 财经数据，拼接上下文 | `urllib` |
| 研判层 | `call_agnes()` + `SYSTEM_PROMPT` | 大模型综合研判（漏斗式策略） | Agnes OpenAI 兼容 API |
| 回填层 | `_fetch_tencent_stocks()` | 用模型给的 code 反查实时行情，保证硬数字真实 | `urllib` |
| 组装层 | `assemble()` / `fallback_payload()` | 拼装标准 schema 看板；模型失败则降级 | json |
| 清单层 | `write_manifest()` | 扫描历史看板，生成前端枚举清单 | json |
| 展示层 | `dashboard/index.html` | 纯静态看板（日期切换 + 详情渲染） | 原生 HTML/JS/CSS |

### 1.3 关键设计原则（硬约束）

1. **数字不编造**：所有行情数字（指数、个股价格、涨跌幅、市值、涨跌家数、涨停数）必须来自真实接口或 Web 资料；缺失即填 `null` 并标注。
2. **判断可追溯**：每个 AI 判断字段都带 `source`（数据来源 URL）和 `reasoning`（2-5 句因果链）。
3. **结构化回填优先**：指数行情从真实接口直接结构化回填（不经模型回读），避免模型"丢失"数字。
4. **降级不中断**：Agnes 调用失败 → 写"数据不可用"降级看板 + 保留 `web_context.txt` 原文，流水线不中断。
5. **风控约束**：不做打板、不追涨停；中军候选只选"未涨停、位置好、趋势不破"的容量票。
6. **免责声明**：仅供研究参考，不构成投资建议。

## 二、GitHub Actions 完整流程（含代码级细节）

### 2.1 Workflow 配置（`.github/workflows/daily-brief.yml`）

| 配置项 | 值 | 说明 |
|---|---|---|
| 触发-定时 | `cron: "35 8 * * 1-5"` | UTC 08:35，对应北京时间 16:30（周一至周五） |
| 触发-手动 | `workflow_dispatch` + `trade_date` 输入 | 支持指定交易日重跑 |
| 权限 | `contents: write` / `pages: write` / `id-token: write` | 提交数据 + 部署 Pages |
| 并发控制 | `group: stock-brief`, `cancel-in-progress: false` | 同组串行，避免数据竞争（不取消进行中任务） |
| 运行环境 | `ubuntu-latest`, Python 3.11 | 标准 CI 环境 |

### 2.2 流水线步骤（step by step）

1. **Checkout**（`actions/checkout@v4`，`fetch-depth: 0`）：拉全量历史，便于 git commit 数据。
2. **Set up Python**（`actions/setup-python@v5`，3.11）。
3. **Install deps**：`pip install -r requirements.txt`（当前 requirements.txt 为空注解文件，实际仅用标准库，安装无副作用）。
4. **Generate brief**：注入 Secrets 环境变量后运行脚本：
   - 有 `trade_date` 输入 → `python .../web_brief.py --trade-date "..."`
   - 否则 → `python .../web_brief.py`（默认最近交易日）
   - 注入变量：`AGNES_API_KEY`、`AGNES_BASE_URL`、`AGNES_MODEL`、`BOCHA_API_KEY`、`BRIEF_ROOT=.../stockexpert-daily-brief`
5. **Commit data**：`git add` 看板 data 目录 + manifest.json；若 `git diff --cached` 为空则跳过，否则 commit `chore: 每日看板 YYYY-MM-DD` 并 push。
6. **Setup Pages**（`actions/configure-pages@v5`）。
7. **Upload Pages artifact**（`upload-pages-artifact@v3`，path=`stockexpert-daily-brief/dashboard`）。
8. **Deploy**（`deploy-pages@v4`），输出 Pages URL。

### 2.3 脚本内部执行流程（`web_brief.py`）

```
main()
 ├─ 解析参数 (--dry-run / --trade-date)
 ├─ 计算交易日: 指定日期则校验 is_trade_day；否则 latest_trade_day() 向前回溯到最近交易日
 └─ generate_brief(trade_date, dry_run)
      ├─ build_web_context(trade_date)
      │    ├─ 抓指数 (腾讯优先 → 东财备用) → 结构化 idx_struct + 文本
      │    ├─ 抓新浪行业板块名录 (仅名称, 无涨幅)
      │    ├─ 抓东财盘中快讯
      │    ├─ 博查搜索 × 4 组 query (收评/题材/涨跌家数/板块涨幅)
      │    └─ 返回 (context_text, data_sources, idx_struct)
      ├─ call_agnes(SYSTEM_PROMPT, context)   # 失败返回 None, 带 2 次重试, 超时 120s
      ├─ if raw 为空 → fallback_payload() + 落盘 web_context.txt
      ├─ else 解析 JSON → 取 midcaps code → _fetch_tencent_stocks() 反查实时行情
      │         └─ assemble() 组装最终看板
      ├─ dry_run? 打印预览 : 写 看板.json + write_manifest()
```

### 2.4 失败与降级路径

| 失败场景 | 行为 | 产物 |
|---|---|---|
| Agnes 调用失败（2 次重试均失败） | 写降级看板（`data_quality.overall = "unavailable"`） | 落盘 `web_context.txt` 保留原始抓取 |
| Agnes 返回非 JSON | 异常捕获 → 降级看板 | 仅 Web 上下文 |
| 腾讯指数接口失败 | 自动切换东财 push2；再失败 → `idx_struct` 空，指数填 null | 不阻塞 |
| 博查未配置 `BOCHA_API_KEY` | 跳过搜索，仅用其余源 | 题材/涨跌家数可能缺失 |
| 个股行情反查失败 | `stock_quotes` 空，股价用模型给的兜底（可能为 null） | 标注缺失 |

## 三、选股 / 选板块 / 判断策略（核心审核重点）

### 3.1 方法论总纲：`right-side-core`（右侧交易体系）

全局常量 `METHOD = "right-side-core"`。所有题材与中军的 `methods` 字段均写入该值，表示"右侧交易核心"方法论。

策略漏斗（来自 `SYSTEM_PROMPT`，大模型据此研判）：

```
大盘强度(market_regime.strength)
   ↓
题材四阶段判定(themes[].stage)
   ↓
中军筛选(midcaps[]: 大容量 + 好位置 + 趋势不破)
   ↓
3 日观察区间(suggested_zone + hold_horizon_days=3)
```

### 3.2 第一层：大盘强度判断

**判定维度（`market_regime`）：**

| 字段 | 取值 | 来源 |
|---|---|---|
| `strength` | `strong` / `neutral` / `weak` | AI 综合研判 |
| `indices` | 上证/深成/创业板指 价格+涨跌幅 | 腾讯/东财接口结构化回填 |
| `breadth_up` / `breadth_down` | 涨跌家数 | 博查搜索提取，缺失填 null |
| `limit_up` / `limit_down` | 涨停/跌停家数 | 博查搜索提取，缺失填 null |

**强度推理依据**：模型基于指数表现、权重股动向、炸板率、题材与指数是否脱钩、赚钱效应分化等做因果链判断（见 2026-07-17 实例：`weak`，理由为"权重崩塌、各项数据低迷、炸板率高"）。

### 3.3 第二层：题材四阶段判定

**`themes[].stage` 枚举：**

| 阶段 | 枚举值 | 含义 | 前端配色 |
|---|---|---|---|
| 启动 | `launching` | 新主线首日资金抢筹 | 蓝 `#3498db` |
| 趋势中 | `trending` | 主升浪延续 | 绿 `#2ecc71` |
| 高潮 | `climax` | 一致性强、可能见顶 | 橙 `#e67e22` |
| 退潮 | `fading` | 资金撤离、补跌 | 灰 `#95a5a6` |

**每个题材必带字段：**

- `strength`（`high`/`mid`/`low`）+ `strength_reasoning`
- `stage_reasoning`（2-5 句因果链）
- `catalyst`（催化因素，如"净流入19.3亿首日登顶"）
- `source`（Web 来源 URL）
- `methods`（固定 `right-side-core`）

**实例（2026-07-17 端侧AI）**：`stage=launching`，`strength=high`，催化为"资金首日净流入19.3亿登顶，指数与题材脱钩下的防御性修复方向"。

### 3.4 第三层：中军候选筛选（选股核心）

**中军定义**：大容量（大市值、可容纳大资金）、位置好（未涨停、回调到好买点）、趋势不破（均线/结构未破位）的标的。

**`midcaps[]` 关键字段：**

| 字段 | 含义 | 取值 |
|---|---|---|
| `code` | 6 位代码（必填，可带 sh/sz/bj 前缀） | 模型输出，供反查行情 |
| `position_eval` | 位置评估 | `good`(好位置)/`ok`(可)/`high`(偏高谨慎)/`avoid`(排除) |
| `trend_status` | 趋势状态 | `intact`(完整)/`broken`(破位)/`watch`(观察) |
| `suggested_zone` | 3 日观察区间 `{low, high, reasoning}` | 模型给区间 |
| `hold_horizon_days` | 持有周期 | 固定 `3` |
| `market_cap` / `price` / `change_pct` | 实时硬数字 | 腾讯反查回填（优先于模型值） |

**硬性风控（写入 System Prompt）：**
> 不做打板、不追涨停；中军候选只选"未涨停、位置好、趋势不破"的容量票。

**筛选链路**：题材 → 模型按 `theme_id` 关联中军 code → `related_midcap_codes` 关联回题材 → 反查实时行情验证价格/市值。

### 3.5 第四层：3 日观察区间

`hold_horizon_days = 3` 是全局默认持有周期。模型为每只中军给 `suggested_zone.{low, high}` 并附 `reasoning`（如"12.5~14.0，回踩5日线不破介入"）。前端以 `区间 low ~ high` 呈现。

### 3.6 策略执行的关键约束小结

- 模型**只做研判/综合**，原始数字来自 Web（防止幻觉）。
- 指数/个股行情**结构化回填**，绕过模型回读丢失风险。
- 所有选择（强/弱、阶段、位置、区间）都**必须带 source + reasoning**，审核可追溯。
- 缺失即 `null`，绝不编造——这是策略可信度的基石。

## 四、数据契约（dashboard-contract / 看板.json Schema）

```jsonc
{
  "schema_version": "1.0",
  "trade_date": "YYYY-MM-DD",
  "generated_at": "ISO8601",
  "is_demo": false,
  "market_regime": {
    "strength": "strong|neutral|weak",
    "strength_source": "ai-synthesis",
    "strength_reasoning": "因果链",
    "indices": [{"name","code","price","change_pct","source","fetched_at"}],
    "breadth_up": int|null, "breadth_down": int|null, "breadth_reasoning": str,
    "limit_up": int|null, "limit_down": int|null
  },
  "themes": [{
    "id","name","stage":"launching|trending|climax|fading",
    "stage_source","stage_reasoning","strength":"high|mid|low",
    "strength_reasoning","methods":["right-side-core"],"signals":[],
    "related_midcap_codes":[],"catalyst","source","notes"
  }],
  "midcaps": [{
    "code","name","theme_id","market_cap","price","change_pct","ma5",
    "quote_source","position_eval":"good|ok|high|avoid",
    "position_reasoning","trend_status":"intact|broken|watch",
    "trend_reasoning","suggested_zone":{"low","high","reasoning"},
    "hold_horizon_days":3,"methods":["right-side-core"],"source","notes"
  }],
  "data_sources": [{"type":"web","url","fetched_at"}],
  "data_quality": {"overall":"partial|complete|unavailable","missing":[]},

}
```

`manifest.json` 由 `write_manifest()` 扫描 `dashboard/data/*/*/看板.json` 生成，供前端枚举（含 `trade_date`、`path`、`regime`、`theme_count`、`is_demo`）。

## 五、前端展示（dashboard/index.html）

- **纯静态**，无构建、无后端、无框架，原生 HTML/CSS/JS。
- **布局**：左侧栏日期列表（按 manifest 倒序），右侧主区。
- **渲染逻辑**：
  - `loadAll()`：`fetch('manifest.json')` → 生成侧栏 → 默认选中最新一天。
  - `render(day)`：`fetch(day.path)` 读取看板 → 渲染大盘强度+指数、题材卡片（阶段徽章配色）、中军表格（位置/趋势标签）、数据来源与质量。
  - 颜色映射：`up=绿/down=红`；阶段徽章 `launching/trending/climax/fading` 不同底色；位置标签 `good/ok/high/avoid` 不同底色。
- **容错**：manifest 缺失/空 → 显示提示；单日读取失败 → 显示错误块。
- `cache: "no-store"` 避免 CDN 缓存导致看不到最新。

## 八、审核关注点 / 风险与改进建议

### 8.1 风险点

1. **降级态强度误标**：模型失败时 `strength` 仍为 `neutral`，建议失败时统一置为 `"unknown"` 或独立标记，避免误导。
2. **交易日历简化**：`HOLIDAYS = set()` 为空，仅过滤周末，未排除法定节假日（如春节、国庆），非交易日会生成空/降级看板。生产应接交易所日历 API。
3. **腾讯接口编码**：CI 下偶发乱码导致指数全 null（2026-07-17 实例），需加 `utf-8`/`gbk` 容错解码或字段校验。
4. **资料可靠性依赖博查**：题材研判质量高度依赖博查搜索命中质量；未配置 `BOCHA_API_KEY` 时题材可能全缺。
5. **大模型可控性**：阶段/位置/区间完全由模型主观研判，存在模型波动；建议对关键字段做规则校验（如 `suggested_zone.low < high`）。
6. **免责与合规**：当前为研究参考定位，需确保 Pages 公开页面保留风险提示。

### 8.2 改进建议

- 补全 `HOLIDAYS` 或接交易所日历；非交易日直接 skip 不生成。
- 指数解析增加编码容错与数字校验。
- 降级看板 `strength` 明确为 `"unknown"`。
- 对中军 `suggested_zone`、阶段枚举做后端 schema 校验（jsonschema）。
- 增加 `data_quality` 自动告警（如 `partial` 持续多日则通知）。
- 考虑缓存历史题材阶段，做跨日"阶段演进"对比（当前为单日快照）。
