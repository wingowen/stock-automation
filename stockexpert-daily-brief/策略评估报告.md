# 选股策略评估报告：StockExpert `right-side-core`（右侧交易体系）

> 评估对象：`stockexpert-daily-brief` 项目中的选股/选板块研判策略
> 评估基准日：2026-07-18（已读取 `web_brief.py` 全文、`项目审核文档.md`、2026-07-16/17 实际看板数据）
> 评估框架：策略概述 → 合理性分析（理论支撑 / 共线性冗余 / 参数 / 成本流动性市场环境 / 过拟合前视偏差）→ 优化建议 → 评分卡
> 重要前提：本策略是**定性 LLM 研判框架**，非可回测的量化因子模型，故部分维度（如多重共线性）需按"概念冗余/逻辑冲突"重新解读。

---

## 一、策略概述

### 1.1 本质定位
不是"因子打分→排序→持仓"的量化策略，而是一套**漏斗式研判流水线**：每天收盘后抓公开 Web 财经信息，由 LLM（Agnes）综合研判，产出"大盘强度 + 题材阶段 + 中军候选 + 3日观察区间"的结构化看板。硬数字（指数/个股行情）从腾讯、东财接口**结构化回填**，绕过模型回读，避免幻觉。

### 1.2 四层漏斗逻辑
| 层 | 维度 | 取值 | 作用 |
|---|---|---|---|
| 1 大盘强度 | `market_regime.strength` | strong / neutral / weak | 定总仓位姿态 |
| 2 题材阶段 | `themes[].stage` | launching / trending / climax / fading | 判定主线所处生命周期 |
| 3 中军筛选 | `midcaps[]`（大容量 + 好位置 + 趋势不破） | position_eval / trend_status | 选具体容量标的 |
| 4 观察区间 | `suggested_zone.{low,high}` + `hold_horizon_days=3` | 模型给区间 | 给 3 日参考买点 |

### 1.3 选股条件（硬性约束，写入 System Prompt）
- **不追涨停、不打板**；中军候选只选"未涨停、位置好、趋势不破"的容量票。
- 中军定义 = 大市值/可容纳大资金 + 回调到好买点 + 均线结构未破位。

### 1.4 买卖规则（注意：此处是核心缺口）
- **买入**：模型基于 `position_eval=good` + `suggested_zone` 给出"3日观察区间"，无明确触发价/分批规则。
- **卖出/止损**：仅有"趋势不破"作为软性退出条件 + 固定 `hold_horizon_days=3`，**无显式止损线、无仓位管理、无组合层风控**。
- 结论：**这是"研究信号生成器"，不是完整的可交易系统**。

### 1.5 关键参数
`temperature=0.3`、`hold_horizon_days=3`、`HOLIDAYS=set()`（空）、腾讯/东财/新浪/博查多源采集、模型失败降级。

## 二、合理性分析

### 2.1 选股逻辑是否有理论/实证支撑 —— ⚠️ 逻辑有渊源，实证为零
- **有理论根基的部分**：
  - "右侧交易"（趋势确认后介入而非抄底）对应趋势跟随（trend-following）经典范式，Jegadeesh & Titman 动量研究、切向均线系统等均有文献支撑。
  - "题材四阶段"（启动/趋势/高潮/退潮）对应**主题生命周期**与**行业轮动**实证研究，逻辑自洽。
  - "中军=大容量流动性锚"思路正确——承接大资金、规避小票流动性枯竭，是合理的风控取向。
- **实证严重缺失**：
  - 整个框架**没有任何回测、没有任何与基准（如沪深300/中证全指）的对比**，无法证明"right-side-core 选出的题材/中军"未来 3 日确实跑赢。
  - 两层样本（07-16/07-17）`midcaps` 均为空 → **选股核心从未被实际验证过**，目前只是"题材点评生成器"。
  - 评分：**2/5**（理论可解释，但"策略有效性"未被任何数据证实）。

### 2.2 筛选指标是否存在多重共线性/冗余 —— ⚠️ 概念冗余明显
- 传统 VIF 不适用（非多因子数值模型），但**概念层冗余**存在：
  - `position_eval`（good/ok/high/avoid）与 `trend_status`（intact/broken/watch）高度重叠——两者都在编码"技术面是否健康"，正交性不足，易让模型重复判断同一件事。
  - 矛盾未消解：07-17 实例中 `market_regime=weak` 但 `theme=launching, strength=high`。**弱市 + 强势启动题材**是真实常见场景，但框架对"层1与层2冲突时如何处置"无任何规则（例如弱市下是否应收缩区间、降低评级）。模型自己编造了"指数脱钩下的防御性修复"来圆，说明规则存在真空。
  - 评分：**3/5**（分层清晰，但层间冲突无仲裁、相邻字段冗余）。

### 2.3 策略参数设置是否合理 —— ⚠️ 部分任意，存在运行级 bug
- `hold_horizon_days=3`：对短线波段合理，但**无任何敏感性分析或依据**（3日 vs 5/10日孰优未知）。
- `temperature=0.3`：结构化 JSON 输出下合适。
- **`HOLIDAYS=set()` 是真实缺陷**：仅过滤周末，春节/国庆等非交易日仍会触发 CI、生成空/降级看板，浪费运行且污染数据。
- 阶段切换**无量化阈值**（如涨幅/成交/涨停数到什么程度算 climax），全凭模型主观，结果不可复现。
- `suggested_zone.low/high` 无校验（可能 low>high）。
- 评分：**3/5**（主参数可接受，但节假日 bug + 缺校准是实伤）。

### 2.4 是否考虑交易成本、流动性风险及市场环境 —— ✅ 流动性有，成本缺
- **流动性风险：有考虑（加分项）**。中军强制"大容量"，天然规避小票流动性陷阱。
- **市场环境：有考虑（加分项）**。层1 大盘强度即是市场状态感知。
- **交易成本：完全缺失（减分项）**。无佣金、滑点、买卖价差建模；3日高频换仓下成本对净收益影响可观，却从未计入。
- **组合层风控：缺失**。无仓位上限、无单标的暴露限制、无最大回撤约束、无止损。
- 评分：**3/5**（流动性+市场环境意识到位，但成本与组合风控空白）。

### 2.5 是否存在过拟合风险或前视偏差 —— ✅ 前视基本规避，过拟合换了一种形式
- **前视偏差：基本规避（加分项）**。16:30 收盘后跑、用当日收盘数据推"未来3日区间"，时序正确；未使用未来信息。
  - *微小隐患*：博查搜索可能抓到盘后叙事（提及盘后事件），间接污染"当日研判"，建议对搜索结果做时间窗裁剪。
- **过拟合：传统过拟合不存在（无参数拟合），但存在"LLM 叙事过拟合"**：
  - 模型可用任意故事解释任意走势，难以证伪；
  - **输出不稳定**：同日输入、不同次调用可能给出不同 stage/区间（temperature 非零 + 无规则锚定）；
  - 07-17 把"虎扑论坛复盘"当机构资金面依据、"净流入19.3亿"强断言却源自论坛帖——**强结论 + 弱信源**，是典型的"看似有支撑实则脆弱"。
- 评分：**4/5**（时序合规好，但模型不可复现性与信源脆弱是结构性弱点）。

### 2.6 附加维度：数据质量与可追溯性（本策略的真正命门）—— ⚠️ 设计好，落地弱
- **设计层面优秀**：`source + reasoning` 全字段可追溯、"缺失即 null 不编造"、硬数字结构化回填——工程纪律值得肯定。
- **落地层面脆弱**：
  - 07-17 腾讯接口乱码 → 三大指数 `price/change_pct` 全 null，硬数字可靠性在 CI 下打折；
  - 实质内容高度依赖博查搜索命中质量，未配 `BOCHA_API_KEY` 时题材可能全缺；
  - 信源混杂（搜狐/凤凰/网易/虎扑论坛并列），无信源可信度分级；
  - **降级 bug**：07-16 模型失败却写 `strength=neutral`（应为 `unknown`），`项目审核文档` 已自曝此问题。
- 评分：**3/5**（机制好，但信源质量与降级标注需补强）。

## 三、优化建议（具体、可落地）

### 3.1 把"信号"升级为"可验证策略"（最高优先级）
- 建**历史回放校验**：对每个被选题材，记录其 stage 与之后 3/5/10 日对应行业指数/ETF 收益，滚动统计胜率与超额——哪怕是最简版，也能回答"这框架到底有没有用"。
- 给 `midcaps` 增加**最小命中要求**与**回填成功率监控**，避免连续空仓却无告警。

### 3.2 补齐逻辑真空与冲突仲裁
- 增加"层间冲突规则"：如 `market_regime=weak` 时，仅保留 `launching` 且 `strength=high` 并带资金确认的题材，且自动收缩 `suggested_zone` 宽度、下调 `position_eval` 上限。
- 合并或正交化 `position_eval` / `trend_status`：建议改为"位置分（估值/涨幅分位）+ 趋势状态（均线/结构）"两条独立轴，消除冗余。

### 3.3 参数与运行健壮性
- 补全 `HOLIDAYS` 或接入交易所日历 API；非交易日直接 skip，不生成。
- 指数解析加 `utf-8/gbk` 容错解码 + 数值校验，乱码即标 null 而非写入错误值。
- 后端加 schema 校验：`suggested_zone.low < high`、`stage` 枚举合法、`strength` 与 `market_regime` 一致性。
- 降级看板 `strength` 明确为 `"unknown"`，杜绝失败态误标。

### 3.4 成本与风控补全（若定位为可交易）
- 在 `suggested_zone` 之外增加**止损线**（如跌破区间下沿/某均线即离场）与**仓位建议**。
- 在回放中计入佣金+滑点，报告"扣除成本后"的净胜率。
- 增加 `data_quality` 连续 `partial` 多日告警。

### 3.5 信源治理
- 对 `data_sources` 做**可信度分级**（API硬数据 > 权威财经媒体 > 论坛/自媒体），前端标注；
- 强断言（如"净流入19.3亿"）必须回链到原始数据源并标注置信度，禁止论坛帖充当机构资金面证据；
- 博查搜索结果做时间窗裁剪，规避盘后叙事泄漏。

### 3.6 跨日演进（增值）
- 缓存历史题材 stage，做"阶段演化"对比（launching→trending 占比、退潮预警），从单日快照升级为趋势跟踪。

## 四、评分卡（1–5，5 最佳）

| 维度 | 评分 | 一句话理由 |
|---|---|---|
| 理论/实证支撑 | 2 | 趋势/轮动有文献根基，但零回测、选股核心从未跑通 |
| 指标共线性/冗余 | 3 | 分层清晰，但 position/trend 冗余、层间冲突无仲裁 |
| 参数合理性 | 3 | 主参数可接受，节假日 bug + 无校准是实伤 |
| 成本/流动性/市场环境 | 3 | 流动性+市场环境意识到位，成本与组合风控空白 |
| 过拟合/前视偏差 | 4 | 时序合规好，但模型不可复现、信源脆弱 |
| 数据质量/可追溯 | 3 | 机制优秀，落地信源弱、降级标注有误 |
| **总体可行性** | **谨慎（3/5）** | 工程纪律好、思路正确的"研究信号框架"；距"可验证/可交易策略"仍有本质差距 |

**一句话结论**：作为"每日盘面研判 + 轻量筛选"的研究辅助，框架合理、工程扎实，可**谨慎使用**；但必须补齐**回测验证、层间冲突仲裁、成本与止损、信源分级**四块，否则它只是"看起来专业的点评生成器"，不宜直接作为选股依据。
